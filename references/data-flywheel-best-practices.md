## 调研报告：开源社区数据飞轮的最佳实践、场景与流程

### 一、问题定义
- **要解决什么问题？**
  如何在 `ecom-agent`（电商客服 Agent）中构建"数据飞轮"（Data Flywheel），使系统能够通过日常的用户交互数据自动迭代，持续提升服务质量、画像准确度和模型性价比。
- **当前系统的现状和不足？**
  当前系统主要依赖预设的 Prompt、Guardrails 和静态规则，缺乏一个系统化的流程来自动收集线上反馈（如用户的纠正、拦截的失败 Case）并自发地作用于系统优化。

### 二、业界实践（2-3 个参考）
- **参考 A：NVIDIA Data Flywheel Blueprint (2025/2026)**
  - **核心做法**：构建统一的数据编排服务，标准化收集生产日志（Prompt、Response、Feedback），通过持续评估（LLM-as-a-Judge）筛选高质量数据，用于**模型蒸馏**（Model Distillation）。
  - **效果数据**：用积累的高质量产线数据微调小尺寸模型（如 Llama-3.2-1B），在特定任务上可替代 70B 大模型，降低高达 98.6% 的推理成本而不损失准确率。
- **参考 B：电商领域的 AIGQ 推荐架构飞轮 (Taobao 等)**
  - **核心做法**：实时流式摄取用户行为（点击、转化、追问、修改尺码），分离离线训练与在线推理。在线收集隐式反馈，离线使用 RL（强化学习）或 SFT（监督微调）优化多维度意图理解（如兼顾相关性与多样性）。
- **参考 C：AI Engineering Flywheel (2025 现代 LLM 应用栈)**
  - **核心做法**：包含三个核心阶段：评估 (Evaluation) -> 监控 (Monitoring) -> 持续改进 (Continual Improvement)。利用**动态 Few-shot 样本库**：将线上表现差的 Bad Case 经过人工或大模型修正后，存入向量数据库，在后续的 Prompt 中动态召回（RAG），实现"免微调"的快速防御迭代。

### 三、方案对比
| 方案 | 核心思路 | 优势 | 劣势 | 适用性 |
|------|---------|------|------|--------|
| **方案 1：动态 Few-shot 飞轮 (AI Eng)** | 收集优质/修正后的 Case 存入向量库，在 Prompt 中动态召回（RAG）作为 Few-shot。 | 实现简单，见效快，无需训练算力，可快速修复系统幻觉。 | 占用 Context Window，提升单次推理的时间和资金成本。 | ✅ 高度适用（冷启动和 MVP 阶段） |
| **方案 2：画像与规则自适应飞轮 (电商特化)** | 将用户对 Agent 推荐的纠正（如"我不穿L码"）作为惩罚信号，自动调整画像特征置信度或生成覆写规则。 | 深度契合 `ecom-agent` 的服饰导购逻辑，直接提升业务核心指标。 | 需要修改应用层架构，引入画像状态机、版本控制及冲突仲裁。 | ✅ 高度适用（构建核心业务护城河） |
| **方案 3：模型微调/蒸馏飞轮 (NVIDIA)** | 积累大量高质量对话轨迹，微调小尺寸开源模型（如 Qwen 1.5B/7B）替代闭源大模型。 | 长期成本极低，延迟显著降低，数据资产化和隐私保护极高。 | 初期数据量不足，需要搭建完整的评估与 SFT 训练 Pipeline。 | ❌/部分适用（适合项目后期或规模化扩展阶段） |

### 四、推荐方案
- **建议做什么？**
  采用 **"业务侧轻量飞轮" + "提示词动态飞轮"** 的混合策略作为现阶段的重点，同时为**"模型蒸馏飞轮"**打好地基：
  1. **短链路（业务画像飞轮）**：用户的隐式反馈（退换货、尺码修改、态度负面）直接触发 `Profile Store` 的置信度衰减，或触发 `Preference Detector` 记录强制覆写规则（Hard Constraint）。
  2. **中链路（Prompt 动态飞轮）**：捕获被 `Guardrails` 拦截的 Bad Case 和人工客服介入的 Case，修正后入库，作为 LLM 节点的动态 Few-shot，提升 Prompt 的鲁棒性。
  3. **长链路（模型微调，预留数据）**：目前的重点是**定义标准化数据流**：将所有交互（状态、上下文、输出、结果）落盘为标准 JSONL 格式（兼容 OpenAI 格式），不急于微调，只做高价值数据的清洗和归档，为日后的模型蒸馏做储备。
- **为什么选这个方案？**
  高度符合 `ecom-agent` 当前 MVP 阶段缺乏海量训练数据的客观条件。通过业务层和 Prompt 层的轻量级飞轮，可以快速实现系统"越用越聪明"的用户体感；同时通过建立标准化的生产日志规范，以极低的成本实现了数据资产化。
- **实现复杂度估算**：中等（重在日志拦截与状态管理）。
