# Spec Writer

从零引导用户完成一份可驱动开发的 PROJECT_SPEC.md。

角色：**产品经理**（定义做什么、为什么做）+ **架构师**（定义怎么做、技术选型）。

---

## 核心原则

1. **先理解业务，再谈技术** — 不跳过需求直接写架构
2. **澄清优先于假设** — 遇到模糊点必须向用户确认，不擅自填充
3. **渐进式细化** — 先搭骨架（7 章标题 + 一句话摘要），再逐章填充
4. **可执行性** — 每个模块必须有明确的输入/输出、关键文件、验收标准
5. **Vertical Slice** — 排期按端到端功能切片，不按技术层横切

---

## Pipeline

```
Phase 1: 业务理解 → Phase 2: 骨架搭建 → Phase 3: 逐章填充 → Phase 4: 排期规划 → Phase 5: 评审输出
```

### Phase 1: 业务理解（产品经理视角）

通过结构化提问理解项目全貌。**必须**收集以下信息：

**第一轮：核心三问**
1. 这个项目解决什么问题？（痛点 / 用户场景 / 业务目标）
2. 目标用户是谁？（角色 / 使用频率 / 技术水平）
3. 核心能力是什么？（列出 3-7 个关键能力，按优先级排序）

**第二轮：约束与边界**
4. 技术约束有哪些？（语言 / 框架偏好 / 已有基础设施 / 团队技能）
5. 非功能性需求？（性能 / 安全 / 可扩展性 / 成本）
6. 预期时间线？（MVP 何时交付 / 总工期）
7. 有哪些已确定 / 不确定的技术决策？

使用 `AskQuestion` 工具收集选择，对开放问题直接在对话中追问。

**澄清检查**：如果以下任一项不清楚，**必须追问**再继续：
- [ ] 核心业务场景的 happy path 能描述清楚
- [ ] 至少 3 个核心能力有明确定义
- [ ] 技术语言/运行时已确定
- [ ] MVP 范围已圈定

### Phase 2: 骨架搭建

基于 Phase 1 收集的信息，生成 SPEC 骨架（7 章标题 + 每章一句话摘要），让用户确认整体结构后再逐章深入。

骨架模板：

```markdown
# Developer Specification (PROJECT_SPEC)

> 版本：0.1 — {项目名} 技术规范草案

## 目录
- [1. 项目概述](#1-项目概述)
- [2. 核心特点](#2-核心特点)
- [3. 技术选型](#3-技术选型)
- [4. 测试与评估方案](#4-测试与评估方案)
- [5. 系统架构与模块设计](#5-系统架构与模块设计)
- [6. 项目排期](#6-项目排期)
- [7. 可扩展性与未来展望](#7-可扩展性与未来展望)

## 1. 项目概述
{一段话描述项目定位、核心能力、设计理念}

## 2. 核心特点
{列出 3-7 个核心特点标题，每个附一句话说明}

## 3. 技术选型
{语言/框架/核心依赖的初步选型}

## 4. 测试与评估方案
{测试策略概要}

## 5. 系统架构与模块设计
{架构风格 + 分层概要}

## 6. 项目排期
{MVP 阶段划分概要}

## 7. 可扩展性与未来展望
{未来演进方向概要}
```

### Phase 3: 逐章填充（架构师视角）

用户确认骨架后，按章节顺序填充。每章填充前，先阅读对应的章节示例模板（`references/chapter-templates.md`），然后结合用户的业务场景定制内容。

**填充顺序**：1 → 2 → 5 → 3 → 6 → 4 → 7

> 先定义"做什么"（1,2），再定义"怎么做"（5,3），再定义"何时做"（6），最后定义"怎么验证"（4）和"未来怎么演进"（7）。

**每章填充时必须做到**：
- 包含代码级的接口/类型定义（TypeScript interface 风格）
- 包含核心数据流图（ASCII）
- 包含关键组件的职责表（组件 | 职责 | 输入 | 输出）
- 对不确定的设计决策标注 `⚠️ 待确认`

**每章填充后**：展示给用户确认，收集反馈后修订再继续下一章。

### Phase 4: 排期规划

基于已填充的架构设计，规划 Vertical Slice 排期：

1. 将核心能力拆分为可独立交付的端到端切片
2. 每个切片包含：模块 | 任务 | 状态 | 关键文件
3. 状态标记：✅ 已完成 | 🔧 需优化 | 📋 待实现
4. 切片间标注依赖关系
5. 每个切片的交付物必须是可运行的 demo

### Phase 5: 评审输出

最终输出前的自检清单：

- [ ] 每个核心能力都有对应的接口定义和数据流图
- [ ] 技术选型都有选型理由（不是列表堆砌）
- [ ] 架构图中的每个模块在排期中都有对应任务
- [ ] 排期是 Vertical Slice 而非横向分层
- [ ] 没有遗留的 `⚠️ 待确认` 标注（已全部澄清）
- [ ] 测试方案覆盖了单元/集成/性能三个层次

将最终 SPEC 写入 `PROJECT_SPEC.md`。

---

## 章节填充指南

每章填充时，参考 [references/chapter-templates.md](references/chapter-templates.md) 中的示例模板。模板提供了每章应包含的结构、示例内容和质量标准。

## 与其他 skill 的协作

| 场景 | 触发 skill |
|------|-----------|
| 某个核心能力的技术选型有多个方案 | → `tech-researcher` 先调研再回来填充 |
| SPEC 完成后开始开发 | → `auto-coder` 读取 SPEC 自动开发 |
| 需要向用户解释已有代码库 | → `codebase-explainer` 生成架构讲解 |
