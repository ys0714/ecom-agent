## 核心原则

1. **先调研，再决策，最后编码** — 避免闭门造车
2. **搜索真实证据** — 不靠预训练知识猜测，用 WebSearch 搜索最新实践
3. **对齐业务场景** — 不照搬通用方案，结合本项目的具体约束（服饰默选规格推荐 + 电商客服）
4. **输出可执行建议** — 不是学术综述，而是"我们应该做什么 + 为什么 + 怎么做"

## Pipeline

```
理解问题 → 搜索证据 → 分析对比 → 生成报告 → 澄清问题 → 用户确认 → 更新 SPEC
```

### Step 1: 理解问题

明确要调研什么：
- 用户提出的需求/技术方案是什么？
- 当前 SPEC 和代码中这个模块的现状是什么？
- 核心决策点是什么？（技术选型？算法设计？架构模式？）

读取相关 SPEC 章节和代码文件，建立 baseline。

### Step 2: 搜索证据

使用 WebSearch 搜索 **3-5 个维度**：

| 搜索维度 | 示例搜索词 | 目的 |
|---------|-----------|------|
| **开源高星项目** | `"{topic}" open source github stars 2025` | 找到参考实现 |
| **行业头部实践** | `Amazon/Google/Meta "{topic}" production system` | 了解规模化验证的方案 |
| **学术前沿** | `"{topic}" arxiv 2025 best practices` | 捕捉最新方法论 |
| **工程博客** | `"{topic}" engineering blog architecture design` | 获取落地经验 |
| **电商/推荐领域** | `ecommerce "{topic}" recommendation` | 聚焦业务场景 |

每个维度至少搜 1 次，总搜索 3-5 次。

### Step 3: 分析对比

将搜索结果整理为结构化对比：

```markdown
## 方案对比

| 方案 | 来源 | 核心思路 | 优势 | 劣势 | 适用我们吗 |
|------|------|---------|------|------|-----------|
| A    | ...  | ...     | ...  | ...  | ✅/❌/部分  |
```

关键：不是列举所有搜索结果，而是**筛选出和我们的业务场景相关的 2-3 个方案**进行深度对比。

### Step 4: 生成调研报告

报告结构（严格遵循）：

```markdown
## 调研报告：{主题}

### 一、问题定义
- 要解决什么问题？
- 当前系统的现状和不足？

### 二、业界实践（2-3 个参考）
- 参考 A：{来源} — {核心做法} — {效果数据}
- 参考 B：...

### 三、方案对比
| 方案 | 核心思路 | 优势 | 劣势 | 适用性 |

### 四、推荐方案
- 建议做什么？
- 为什么选这个方案？
- 实现复杂度估算（高/中/低）

### 五、待澄清问题
- 需要用户确认的决策点（列出 2-3 个）

### 六、优先级建议
| 优化项 | 业务价值 | 复杂度 | 建议（P0/P1/P2） |
```

### Step 5: 澄清问题

报告生成后，**必须向用户提出待澄清的问题**，例如：
- "你更倾向方案 A 还是方案 B？"
- "这个场景在你的业务中实际频率有多高？"
- "需要支持 XX 特性吗？还是 MVP 阶段可以简化？"

使用 `AskQuestion` 工具收集用户选择。

### Step 6: 更新 SPEC

用户确认后，将调研结论更新到 `PROJECT_SPEC.md` 的对应章节，然后同步：

```bash
python3 .claude/skills/auto-coder/scripts/sync_spec.py --force
```

**只有 SPEC 更新完成后，才能进入编码阶段。**

## 调研质量检查

报告输出前，自检以下条件：

- [ ] 至少搜索了 3 个不同来源（不全是同一个网站）
- [ ] 引用了至少 1 个 2025 年的参考（确保时效性）
- [ ] 对比了至少 2 个方案（不是只推荐一种）
- [ ] 每个推荐都有"为什么适合我们"的理由（不是泛泛而谈）
- [ ] 列出了待澄清问题（不是直接替用户做决定）
- [ ] 给出了优先级和复杂度估算（可执行，不是空谈）

## 与 auto-coder 的协作

当 auto-coder 准备开发新功能时，如果该功能涉及以下情况，**必须先触发 tech-researcher**：

- 新的算法或数据结构设计
- 新的外部依赖/框架引入
- SPEC 中标注为"参考开源社区"的模块
- 用户明确要求"先调研"
- 涉及用户体验/转化率的功能（如推荐解释性、话术策略）

调研完成 + 用户确认 + SPEC 更新 → 才能进入 auto-coder 的 Implement 阶段。
